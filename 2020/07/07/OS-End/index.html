[持久化]I/O设备; 总线, DMA, GPU

沟通计算机&物理世界。
CPU视角：能与CPU交换数据的接口(Status(rw), Command(rw), Data(ro)寄存器)。
I/O设备以导线的方式（某种接口协议）连接到计算机上
CPU：port I/O(in/out)；memory-mapped I/O。CPU就是和设备控制器交互！

操作系统抽象I/O，为应用提供接口？cd /dev

键盘控制器
IBM/PC，“硬编码”到两个I/O port: 0x60(data, ro)，0x64(status/command, rw)

磁盘控制器
ATA controller，primary: 0x1f0(data) — 0x1f7(status); secondary: 0x170 — 0x177

中断控制器
CPU有一个中断引脚，收到某个特定的电信号会触发中断
保存到cs, rip, rflags, ss, rsp
跳转到中断向量表对应项执行

Intel 8259 PIC(programmabable interrupt controller)
其他设备可以向中断控制器连线，中断控制器连接到CPU
中断屏蔽……
slave->master->cpu

总线：一个特殊的I/O设备

“另一个地址空间”；把cpu的地址和数据转发到IO设备上
IO<->总线<->cpu，每次只要把端口对应的线路选通即可传输数据
IBM PC的CPU其实只看到这一个IO设备

总线可以桥接其他总线(PCI->USB)

lspci -tv 查看I/O设备

lsusb -tv

PCI总线

Direct Memory Access(DMA)

CPU一个Byte一个Byte搬数据(e.g. 内存->网卡)浪费

在系统里增加一个新CPU，专门执行”memcpy”程序（没必要使用复杂的通用寄存器实现它）

数据一致性

内存一致性：CPU直连DRAM, PCIe, DMI. CPU带PCIe lanes: 保证cache和DMA数据的一致性

GPU: 既是I/O设备，也是处理器

超高性能并行处理器(OpenGL, CUDA)；CPU更偏向于优化顺序指令执行

DRAM(CPU)→DRAM(GPU), instr(CPU)→GPU ==> res(GPU)→DRAM(GPU)→DRAM(CPU)

cudaMalloc(),cudaMemcpy(), cudaMemcpyHostToDevice(), cudaMemcpyDeviceToHost()……

应用程序不可以直接访问I/O设备，操作系统需要对设备进行抽象（虚拟化）。
I/O设备的抽象：对设备(文件描述符)的I/O
I/O设备可以理解成字节的序列

Rendering Pipeline

设备驱动程序：对设备进行一定管理，对上层提供统一的接口（“面向对象”访问I/O，设备=支持read,write,ioctl,……的对象）
设备驱动程序把HALAPI调用(read, write, ioctl)翻译成设备驱动程序，未必一定要有物理设备！cat /dev/zero(urandom) | head -c 512 | xxd（取前512字节并以16进制打印）

fflush(), lseek()

[持久化]设备驱动程序

I/O设备的抽象

应用程序有访问I/O设备的要求(printf希望输出字符到屏幕……)

应用程序直接访问I/O设备的后果(无法写出portable的代码，否则你要分别对不同的设备编程；应用之间需要互斥、同步、协商设备资源的使用)

所以OS要对设备抽象（虚拟化）

I/O设备虽然复杂，但几乎可以理解为字节的序列（流或数组）

在OS中设置一个抽象层，对上层提供统一的接口，以”面向对象“方式访问I/O设备

以同一个接口访问不同的I/O设备，实现接口的代码就是设备驱动程序。

设备驱动程序把HAL API调用(read, write, ioctl)翻译成设备指令

cat /dev/urandom | head -c 512 | xxd，这个虚拟设备直接往缓冲区里写

Linux设备驱动：Everything is a file：设备提供文件操作就好了

fflush()：将缓冲区的数据真正写入os

flush(), fsync()将数据写入IO设备中，一般用fsync()，flush()是设备关闭前的flush

tty：哪个终端

stty：查看终端信息，使用了系统调用ioctl()读取信息
unlocked_ioctl(): 高性能的驱动通过unlocked_ioctl避免锁BKL
compat_ioctl(): 机器字长的兼容性

两类特殊设备：磁盘&网卡

磁盘：以数据块(block)为单位访问；大吞吐量；应用程序一般不直接访问(一般是文件系统；大量并发访问)

网络设备: 速度快；大小不确定；延迟敏感

应用程序眼中的IO设备

磁盘：/dev/中有许多设备文件

strace查看：mmap(), write()……访问设备确实使用了文件系统API

网络设备：另一组API(send, recv, socket……)

[持久化]文件系统API：文件、目录和文件系统管理

OS有将数据安全可靠地“持久化”到存储设备的需求；应用程序直接通过驱动访问存储设备

文件系统：存储设备的虚拟化

磁盘（I/O设备）= 一个RW的字节序列
虚拟磁盘（文件）= 一个RW的动态字节序列
虚拟磁盘管理：虚拟磁盘的名称管理、查找……

文件系统：保存在持久存储上的数据结构（规定了数据在磁盘上的存储方式‘允许对数据结构操作）；可以实现在kernel中，可以是应用程序(fuse)

目录和文件

key-value;

目录：文件/目录的集合（形成一棵树）；相关的数据存在相近的目录（局部性）

文件API

文件描述符：open, dup/dup2, fork……；

void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset): flags取MAP_SHARED将文件映射到地址空间中，MAP_PRIVATE创建一个copy-on-write的副本。进程地址空间与虚拟磁盘建立关系。SIGSEGV: 越权; SIGBUS: 访问非法offset……关于SIGSEGV和SIGBUS具体内容[这篇回答](Linux中的mmap映射 [二] - 兰新宇的文章 - 知乎 https://zhuanlan.zhihu.com/p/71517406)**（注意映射过程中按页对齐的问题！）**

ftruncate()/truncate(): 改变文件大小

文件访问的游标：lseek()

fork后，子进程“浅拷贝”一份offset（so父子进程共享一个offset）；dup同样也是“浅拷贝”offset

O_APPEND方式打开的文件，offset永远在最后（无论是否fork）

目录/文件系统API

mkdir(使用了mkdir()系统调用)：创建一个目录，可设置访问权限

rmdir：删除一个空目录；并不是“递归删除”，rm -rf 会遍历目录，逐个删除

FAT文件系统树结构的叶子节点是目录或者文件；

UNIX采用硬链接（允许一个文件被多个目录引用，目录中仅存储指向文件数据的指针；链接目录，跨文件系统）：目录仍是树结构，文件是虚拟磁盘，不同的目录（叶子节点）可指向同一个文件（虚拟磁盘）。“删除”文件：unlink，将目录与文件之间的链接删掉，文件维护一个引用计数，当计数为0（所有链接都删掉后，文件真正删除）。

软链接(symbolic link): 在文件里存储一个“跳转提示”（类似于快捷方式）

引用这个文件时，去找另一个文件；另一个文件的绝对路径/相对路径以文本形式存储在这个文件里；可跨文件系统，可链接目录；

甚至可以创建软链接的硬链接

套娃（符号链接成环）

所有处理符号链接的程序(tree, find,…)都要考虑递归情况

[持久化]文件系统实现：FAT和ext2

文件系统：磁盘上的数据结构。磁盘是以block为单位访问的；磁盘上实现数据结构没有malloc的支持（只有read_blk, write_blk）

FAT file system

需求：树状目录结构；sys中以小文件为主(几个block以内)；无需支持链接
文件（block的集合）=链表(e.g. block No.100→block No.20→……)，顺序存储了所有的块号；空闲链表的alloc()算法

两种设计
在每个block后存放指针，e.g. 一个512B block(510B data + 2 next block_num)？(×)单纯的lseek需读取整块数据
所有指针集中存储在文件系统某个区域（File Allocation Table）？(√)

FAT-12/16/32(即next指针大小)
磁盘分成“cluster”，文件系统头部BPB(BIOS Parameter Block)标记了cluster大小(BPB里面还有其他信息)
在一段区域(FAT)连续存放指针(12/16/32-bit)

小文件还可，大文件性能不好(从head到tail要很多次)

ext2/UNIX file system

集中存储文件metadata
Fast/slow path：大小文件（类似于多级页表）

ext2 inode: Mode, Owner info, Size, Direct blocks, Indirect blocks……

ext2 目录文件：文件名到inode编号的key-value mapping（inode, rec_len, name_len, file_type, name）

inode在磁盘上连续存储，便于缓存/预取

[持久化]持久数据的可靠性; RAID

内存：一断电数据结构就没了；磁盘：宝贵的持久数据（但是任何存储介质都有失效的可能）。

磁盘：random data corruption(数据块坏了) & faul-stop

随机损坏和FAT备份

任何一个数据块都可能损坏
e.g. FAT文件系统
metadata: 开不了机了。但可以重新扫描、推断恢复
FAT: 就是M5，所有next都丢了，碎片化程度不高还好，高了就gg……
data: 丢失一个文件。
FAT损坏terriable！所以有备份。性能？memory里有FAT的writeback缓存，所以几乎不会影响性能。

RAID: 备胎的虚拟化

Failt MOdel: Fail-Stop
磁盘可能在某个时刻突然彻底无法访问……

Redundant Array od Inexpensive(Indenpendent) Disks(RAID)
把多个（不可靠的）磁盘虚拟成一块非常可靠的磁盘

进程：一个CPU分时虚拟成多个虚拟CPU
虚存：一份内存通过MMU虚拟成多个地址空间
文件：把一个存储设备虚拟成多个虚拟磁盘

RAID：多个→一个

最简单的想法：多个磁盘，备份（像FAT的备份）。
A: s1 s2 s3 s4 B: s1 s2 s3 s4 两块磁盘上的数据构成“镜像”
read(V, blk) 可以从A或B中的任意一个读取（甚至 A s1 s2; B s3 s4 并行读！）
write(V, blk) 将同样的数据写入A、B的同一位置

RAID-0: V0→A0, V1→B0, V2i→Ai, V2i+1→Bi
高性能（容量、速度）；But零容错。

RAID-1: V0→(A0,B0) 就是上面的备份想法；容忍一块盘fail

RAID-10: (V0, V1)→(A0, B0, C0,D0) e.g. A0=B0=V0, C0=D0=V1 AB(CD)各组成一个RAID-1，AB、CD之间是RAID-0
ABCD构成一个条带，四个磁盘中各取一个数据块表示两个虚拟的数据块。

Raid-4:Parity Disk; Error Correction Code(ECC)!

关于奇偶校验码，这里有个简单的def

The simple parity-check code is the most familiar error-detecting code. In this code, a k-bit data word is changed to an n-bit code word where n = k + 1. The extra bit, called the parity bit, is selected to make the total number of 1s in the code word even. Although some implementations specify an odd number of 1s.

(V0, V1, V2, V3)→(A0, B0, C0, D0, E0)

（考虑1bit的情况）V0→A0, V1→B0, V2→C0, V3→D0, E0(parity code)=A0 xor B0 xor C0 xor D0
于是，A0 xor B0 xor C0 xor D0 xor E0 == 0 任何一个盘出错，都可以用其他四个盘的数据将其恢复出来（e.g. C0 fail了，上试两边同时xor C0，得到C0 = A0 xor B0 xor D0 xor E0）

由于每一bit都是独立的，所以其他大小的盘也是一样的。

妙啊！

RAID-5: Rotating Parity

分析RAID-4的性能，e.g.

A B C D E
V0 V1 V2 V3 P
V4 V5 V6 V7 P
V8 V9 V10 V11 P

顺序读V0V11，并行，得到所有磁盘总带宽的80%
Random read: 80%
Sequential write V0V11: 80%
Random write: only change V5.
P = V4 xor V5 xor V6 xor V7
P’ = V4 xor V’5 xor V6 xor V7
P’ = P xor V5 xor V’5
所以需要(read V5, read P)+(write V5, write P)
一块盘的write只有之前的1/2，五块就只有10%(???)

所以

A B C D E
V0 V1 V2 V3 P
V4 V5 V6 Ｐ V7
V8 V9 Ｐ V11 V10

将Parity均匀分布到每个disk上。

Seq read/write: 80%
random read 100%
random write 25%，但至少ｎ块盘可以获得n/4的随机写性能

[持久化]崩溃一致性; FCSK & Journaling

mkdir a/b，所以a/b/.. == a

如果系统不crash，磁盘eventually会帮我们顺序写入
RAID: write(V0)→write(A0) write(B0)，磁盘速度不同，crash不一致
ext2: 追加写入 write(inode) write(bitmap) write(data) 多次写入没有原子性保证
{inode,bitmap,data}, {}: 对
{inode}: inode指向了一个还没有分配的内存——corrupted filesystem
{bitmap}: 好像分配了，但不可用，消失了一个block——dead block
{data}: 往一个未分配的block里面写入——garbage
{inode,bitmap}: 文件丢失——inorrect data
{inode, data}: corrupt filesystem
{bitmap,data}: dead block

File System Checking(FSCK)

根据磁盘已有的信息，恢复出“最可能”的数据结构(e.g.检查inode标记的数据块是否bitmap标记为１；检查是否存在dandling link，没有链接的inode被移到lost+found文件夹中)……

crash→next boot→fsck（扫描磁盘，把filesystem恢复到一致的状态上）那fsck崩溃怎么办（＊＾＿＿＾＊）

Three easy pieces上还提到了一些fsck的一些问题

However, fsck (and similar approaches) have a bigger and perhaps more fundamental problem: they are too slow.……This situation is akin to dropping your keys on the floor in your bedroom, and then commencing a search-the-entire-house-for-keys recovery algorithm, starting in the basement and working your way through every room. It works but is wasteful.

Journaling(or Write-Ahead Logging)

The basic idea is as follows. When updating the disk, before over-writing the structures in place, first write down a little note (somewhere else on the disk, in a well-known location) describing what you are about to do. Writing this note is the “write ahead” part, and we write it to a structure that we organize as a “log”; hence, write-ahead logging.

flush(): 缓冲区内容（flush之前的请求）全部落盘才会返回（sync???）

简化的问题:　append-only的block vector

flush(sync) block1 block2 block3 block4 flush(sync) OK flush(sync)……

写block1,2,3,4时如果crash，第二个flush是不会返回的。反之，如果看到”OK”，则表明第二个flush返回了(commit)，block1,2,3,4落盘了。”OK”就是Journal

BTRFS: everything is a B-Tree

任何一个数据结构都有两种实现方法

FS(Tree…)+History(历史操作序列)，History其实就是append-only的!!!
所以磁盘分成两部分？History(Journal) + data structure，先写history(append-only)，在实际更新数据结构之前就写入journal

TxBegin, id=1 I[v2] B[v2] Db TxEnd, id=1 ……
inode bitmap data block

Journaling优化

批处理(xv6; jbd)：多次系统调用的Tx合并成一个，减少log的大小; jbd：定期write back
Checksum(ext4)：不再标记TxBegin/TxEnd；直接标记Tx的长度和checksum
Metadata journaling：只对inode和bitmap做journaling可以提高性能

xv6：begin_op, end_op之间对文件系统的操作是atomic的！

书上提到的几种Journaling

(Full) Data Journaling

  1. Journal write: Write the contents of the transaction (including TxB, metadata, and data) to the log; wait for these writes to complete.
  2. Journal commit: Write the transaction commit block (containing TxE) to the log; wait for write to complete; transaction is said to be committed.
  3. Checkpoint: Write the contents of the update (metadata and data) to their final on-disk locations.

Making the Log Finite（释放Journal中已经写入磁盘的数据）

  1. Journal write: Write the contents of the transaction (containing TxB and the contents of the update) to the log; wait for these writes to complete.
  2. Journal commit: Write the transaction commit block (containing TxE) to the log; wait for the write to complete; the transaction is now committed.
  3. Checkpoint: Write the contents of the update to their final locations within the file system.
  4. Free: Some time later, mark the transaction free in the journal by updating the journal superblock.

Metadata Journaling（避免data block写两次；避免了指针指向garbage）

  1. Data write: Write data to final location; wait for completion (the wait is optional; see below for details).
  2. Journal metadata write: Write the begin block and metadata to the log; wait for writes to complete.
  3. Journal commit: Write the transaction commit block (containing TxE) to the log; wait for the write to complete; the transaction (including data) is now committed.
  4. Checkpoint metadata: Write the contents of the metadata update to their final locations within the file system.
  5. Free: Later, mark the transaction free in journal superblock.
